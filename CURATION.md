# Riftbook Curation Rules

Riftbook is curated around practical AI-assisted workflows.

A resource should earn its place because it helps a builder understand, build, review, debug, ship, or maintain work with AI agents.

## Acceptance principle

Accept a resource only if it improves one of these areas:

- Project understanding
- Agent context quality
- Code review quality
- Debugging accuracy
- Frontend quality
- Token efficiency
- Workflow repeatability
- Tool setup clarity
- Agent memory or retrieval design
- Learning path quality

If the resource is interesting but does not help a real workflow, keep it out.

## Accept when

A resource is a good fit when it has most of these:

- Clear practical use case
- Official source link
- Working install or usage path
- Strong fit for AI-assisted coding or agent workflows
- Specific prompt, command, or workflow example
- Clear limits and failure modes
- Clear good-fit and bad-fit scenarios
- Evidence that it is maintained or still useful
- No affiliate or tracking links
- No private data

## Defer when

Defer a resource when it is:

- A plain link with no explanation
- A generic prompt with no tested workflow
- Mostly marketing copy
- Broken, abandoned, or unclear
- Duplicative of an existing card without adding value
- Missing official source links
- Missing verification steps
- Too vague for a reader to act on

## Resource tiers

| Tier | Meaning |
|---|---|
| Premium Pick | Strong default recommendation with clear, repeated workflow value |
| Top Selected | High-quality resource for a major use case |
| Core Framework | Architecture-level system that shapes retrieval, memory, review, or agent behavior |
| Use Case Tool | Useful for a narrow but real problem |
| Hot Skill | Practical skill that improves agent output in a visible way |
| Experimental | Interesting, but still needs caution, testing, or clearer limits |
| Needs Review | Present only when work is incomplete and should not be treated as endorsed |

## Evaluation rubric

Score each resource from 0 to 2.

| Area | 0 | 1 | 2 |
|---|---|---|---|
| Workflow value | Unclear | Useful in some cases | Clearly useful in real work |
| Setup clarity | Missing | Partial | Easy to follow |
| Agent usefulness | Generic | Some agent value | Strong agent workflow value |
| Verification | Missing | Basic | Clear checks and expected output |
| Limits | Missing | Some limits | Good fit and bad fit are clear |
| Source quality | Weak | Acceptable | Official or primary source |

Recommended decision:

- 10 to 12: Accept
- 7 to 9: Accept after edits
- 4 to 6: Needs review
- 0 to 3: Defer

## Required questions before adding anything

1. What exact problem does this solve?
2. Who should use it?
3. When should they avoid it?
4. What command, prompt, or workflow proves its value?
5. What can go wrong?
6. What should the agent verify?
7. Where should this live in the repo?
8. Which README files need to be updated?

## Final standard

A Riftbook entry should make the reader think:

> I know when to use this, how to try it, what to avoid, and how to verify the result.

If it does not do that, it needs more work.
